# 电路图智能评价系统 — 项目方案

## 一、项目背景

目前电路图自动评价领域几乎是空白。现有电路仿真软件（Multisim、LTspice）可以模拟电路行为但不提供"评价"；EDA 工具（立创 EDA、Altium）有 DRC（设计规则检查）但针对的是 PCB 布线而非原理图正确性。

**核心思路**：通过图像识别电路图中的元器件、节点和连接关系，转化为结构化文字描述，再由大语言模型（LLM）基于电路知识进行多维度评价——判断电路是否正确、整体质量如何，识别短路等明显错误、电阻阻值过大过小等影响电路质量的缺陷。

## 二、项目目标

构建一个**电路图智能评价系统**，实现：

1. **多源输入支持**：教科书/习题截图、EDA 导出原理图、面包板/实际接线照片
2. **双层输出**：
   - **快速评价**：总体评分 + 致命错误列表（短路、断路等）
   - **详细诊断报告**：错误定位、严重程度、修改建议、元器件参数合理性分析
3. **可扩展评价框架**：先搭建框架，具体评价维度后续根据实验效果迭代

## 三、探索路线

### 阶段一：假设验证（纯手动）

- 人工看图，用文字描述电路（元件类型、参数、连接关系）
- 将文字描述输入 LLM，验证 LLM 能否正确评价电路
- **目标**：确认"文字描述 → LLM 评价"这条链路可行，再投入 CV 模型

### 阶段二：结构化中间表示

- 设计电路描述的结构化 JSON 格式（元件列表、节点列表、连接关系）
- 将自然语言描述转换为结构化数据，减少 LLM 幻觉
- 基于结构化数据设计评价 Prompt

### 阶段三：CV 接入（目标检测）

- 用成熟目标检测模型（YOLO 系列）识别电路图元件
- 自动提取元件参数（阻值、容值等）和连接关系
- 替换人工描述环节，实现端到端自动化

## 四、技术方案对比

### 方案 A：串行流水线（推荐）

```
电路图 → 文字描述 → 结构化中间表示 → LLM多维度评价 → 评分+报告
```

| 层级 | 职责 | 技术选型 |
|------|------|----------|
| 输入层 | 接收多种来源的电路图 | 图像预处理 / 人工标注 |
| 描述层 | 提取元件、参数、连接关系 | 人工（前期）→ YOLO+OCR（后期） |
| 结构化层 | 自然语言 → JSON 结构化表示 | LLM / 规则引擎 |
| 评价层 | 多维度电路评价 | LLM（Claude / GPT-4） |
| 输出层 | 生成评分 + 诊断报告 | 模板引擎 + LLM |

**结构化中间表示示例**：

```json
{
  "components": [
    {"id": "R1", "type": "resistor", "value": "220Ω", "nodes": ["A", "B"]},
    {"id": "LED1", "type": "led", "color": "red", "vf": "2.0V", "nodes": ["B", "GND"]},
    {"id": "V1", "type": "voltage_source", "value": "5V", "nodes": ["A", "GND"]}
  ],
  "connections": [
    {"from": "A", "to": "B", "via": "R1"},
    {"from": "B", "to": "GND", "via": "LED1"},
    {"from": "A", "to": "GND", "via": "V1"}
  ]
}
```

**优点**：
- 每层独立可测，排查问题方便
- 结构化中间表示消除 LLM 幻觉，评价结果更可靠
- 替换 CV 模块时上下游无需改动
- 结构化数据可复用（可视化、仿真对接等）

**缺点**：
- 多一层转换，链路稍长
- 结构化 schema 设计需要迭代打磨

---

### 方案 B：直接端到端

```
电路图 → 自然语言描述 → 单次 LLM 调用（巨型 Prompt） → 输出
```

**实现方式**：把元件描述和所有评价标准塞进一个 Prompt，LLM 一次性输出评分和报告。

**优点**：
- 原型最快，一个脚本就能跑通
- 无需设计中间表示，探索成本低

**缺点**：
- 缺少结构化中间态，长电路描述容易漏判
- Prompt 极长且难以调试，改一个维度可能影响其他
- 后期接入 CV 需要重新设计整个链路
- LLM 对复杂电路的自由文本描述理解不稳定

---

### 方案 C：多 Agent 并行评价

```
电路图 → 文字描述 → 结构化表示 → Agent1(安全检查) ─┐
                                    Agent2(正确性)   ├→ 汇总 → 报告
                                    Agent3(质量评分) ─┘
```

**实现方式**：多个 LLM Agent 分别关注不同评价维度：

| Agent | 职责 | 示例 |
|-------|------|------|
| 安全检查 | 短路、断路、极性、过流风险 | "电源正负极直接短接" |
| 正确性 | 电路逻辑是否实现预期功能 | "LED 应串联限流电阻" |
| 质量评分 | 参数合理性、优化建议 | "R1 阻值偏大，LED 亮度将过低" |

**优点**：
- 每个 Agent 专注单一维度，评价更深入
- 可单独调优某个 Agent 而不影响其他
- 结果天然分层，便于按需展示

**缺点**：
- 调用次数 ×3，API 成本高
- 探索阶段设计偏重
- Agent 间结论可能冲突，需要汇总仲裁逻辑

---

### 方案对比总结

| 维度 | 方案 A（流水线） | 方案 B（端到端） | 方案 C（多Agent） |
|------|:--:|:--:|:--:|
| 原型开发速度 | 中 | 快 | 慢 |
| 评价准确性 | 高 | 中 | 最高 |
| 可维护性 | 高 | 低 | 中 |
| API 成本 | 中 | 低 | 高 |
| 后期扩展性 | 高 | 低 | 高 |
| 适合阶段 | 全阶段 | 快速验证 | 生产优化 |

## 五、推荐实施路径

1. **先从方案 B 的极简版起步**：用 3-5 张典型电路图，人工写描述，单次 LLM 调用验证"LLM 能否评价电路"这个核心假设（1-2 天）

2. **假设验证通过后，切换到方案 A**：设计结构化 JSON Schema，搭建流水线框架，系统化测试（1-2 周）

3. **评价维度成熟后，按需引入方案 C 的并行 Agent**：对关键维度（安全、正确性）使用独立 Agent 深度评价（按需）

4. **最后接入 CV 模块**：用 YOLO + OCR 替换人工描述，实现自动化（待定）

## 六、评价维度规划

框架设计时预留以下评价维度，具体实现按实验效果迭代：

### 硬错误（致命级）
- 短路检测：电源到地直连、元件两端短接
- 断路检测：悬空节点、未闭合回路
- 极性错误：电解电容、二极管、LED 反接
- 电源冲突：多个输出引脚直连

### 正确性（功能级）
- 拓扑检查：电路结构是否可实现声称的功能
- 元件必要性：是否缺少关键元件（如 LED 缺少限流电阻）
- 参数合理性：阻值/容值是否在合理范围

### 质量评估（优化级）
- 功耗分析：是否存在不必要的功耗
- 参数优化：电阻值调整建议、去耦电容建议
- 鲁棒性：是否考虑元件公差、温度影响
- 成本优化：是否有更廉价的替代方案

## 八、风险与对策

| 风险 | 影响 | 对策 |
|------|------|------|
| LLM 对电路理解不准 | 评价结果不可靠 | 结构化中间表示降低歧义；建立评测基准集 |
| 复杂电路描述过长 | 超出 LLM 上下文 | 分模块评价；优先支持中小规模电路 |
| 电路图种类繁多 | CV 模型泛化差 | 先聚焦 2-3 种常见电路类型 |
| 缺乏标注数据 | 无法训练 CV 模型 | 先用 LLM 辅助标注；公开数据集迁移学习 |
