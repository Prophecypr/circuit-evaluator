## 1. 系统概述

电路图智能评价系统支持**图像输入**和**文本输入**两种方式对电路设计进行自动评审。

```
输入                 处理                              输出
─────               ─────                              ─────
电路图片 → YOLO检测元件 + CRNN读数值 + 接线推理 + LLM评审 → 标注图+TXT报告
文字描述 → LLM结构化转换 + 规则检查 + SPICE仿真        → JSON报告
```

**技术栈**: Python 3.13, PyTorch, OpenCV, YOLOv8n (Ultralytics), Pydantic, OpenAI SDK

**外部依赖**: DeepSeek API (LLM), Tesseract 5.5 (IC型号OCR), Ngspice (SPICE仿真, 可选)

## 3. 模型清单

### YOLOv8 检测模型 (基于 yolov8n.pt, ~6MB each)

| 模型 | 路径 | 类别 | 性能 | 用途 |
|------|------|:----:|------|------|
| CGHD 56类 | `runs/detect/cghd_56cls/weights/best.pt` | 56 | mAP50=68.9% | **主模型**: 元件+节点+文字检测 |
| HCD 17类 | `runs/detect/circuit_real/weights/best.pt` | 17 | mAP50=98.5% | 备选元件检测 |
| 文字检测 | `runs/detect/circuit_text/weights/best.pt` | 1 | mAP50=99.4% | 数值文字区域检测 |
| 端口检测 | `runs/detect/circuit_port/weights/best.pt` | 4 | mAP50=98.9% | 端口点检测(论文用) |

### CRNN OCR 模型

| 模型 | 路径 | 字符集 | 性能 | 用途 |
|------|------|--------|------|------|
| 手写OCR | `runs/ocr_crnn/best.pt` | 28字符 | val_loss=0.084 | 默认: 手写电路数值识别 |
| 机画OCR | `runs/ocr_crnn_machine/best.pt` | 28字符 | val_loss=0.164 | 机画图优化版 |

**CRNN架构**: CNN(5层, 64→128→256→512通道) → BiLSTM(2层, 256隐藏) → FC → CTC解码

**字符集**: `0123456789.kKmMΩμunpFV AHz-+` (28个)

**OCR后处理**: 
- 简写展开: `3V3`→`3.3V`, `4R7`→`4.7Ω`
- 混淆修正: `H`→`kΩ`, `V`→`μF`, `z`→`kΩ`
- 上下文修正: 裸数字`33`近V-DC → `3.3V`
- Tesseract回退: IC元件用Tesseract读芯片型号

### CGHD 56类检测能力

| 类别 | 包含 |
|------|------|
| 无源元件(8) | 电阻/可调电阻/光敏电阻, 无极性电容/极性电容/可调电容, 电感/铁氧体电感 |
| 有源元件(11) | 二极管/LED/稳压管/防雷管, 晶闸管/双向晶闸管/触发二极管, 压敏电阻, BJT/FET/光电晶体管 |
| 电源(5) | DC电压源/AC电压源/电池, GND/VSS |
| 运放/IC(6) | 运放/施密特, 光耦, 通用IC/NE555/稳压器 |
| 逻辑门(6) | XOR/AND/OR/NOT/NAND/NOR |
| 机电(9) | 开关/继电器/插座/保险丝, 扬声器/电机/灯, 麦克风/天线 |
| 其他(6) | 变压器/耦合电感, 晶振/机械, 探头x3 |
| 辅助(5) | text/junction/crossover/terminal(检测时跳过) |

## 4. 图像检测流水线

### 4.1 处理流程 (`src/vision/unified_pipeline.py`)

```
输入图片
│
├─ Step 1: YOLO CGHD 56类检测
│   ├ 过滤非元件类 (text/junction/crossover/terminal/probe)
│   ├ NMS去重 (同族重叠合并, IoU>0.35)
│   ├ P0: GND上下文 → V-AC自动修正为V-DC
│   └ 输出: 元件列表 + 节点列表
│
├─ Step 2: HCD circuit_text 文字区域检测
│   └ 过滤GND符号内误检
│
├─ Step 3: CRNN OCR识别
│   ├ 预处理: 等比缩放32高, 填充128宽, 反色归一化
│   ├ CTC贪婪解码
│   ├ _clean_ocr: 简写展开 + 双点修正
│   ├ _ocr_variants: 候选解读生成 (H/k, V/μF, z/kΩ)
│   └ 3c: Tesseract读取IC型号 (扫描IC框±20px)
│
├─ Step 4: 数值匹配 (距离优先)
│   ├ 筛选: 距离≤250px + 模式验证 + 反模式过滤
│   ├ 变体: 候选值尝试匹配
│   └ 上下文: 裸数字"33"近V-DC→"3.3V"
│
├─ Step 5: 设计位号 (R1, C1, LED1...)
│
├─ Step 6: 端口计算
│   ├ Sobel边缘分析 → 判断水平/垂直朝向
│   ├ GND永不旋转, 其余按Sobel结果旋转
│   └ 归一化坐标转绝对像素坐标
│
├─ Step 7: 接线推理
│   ├ 节点处理: 网格捕捉(5px) → 并查集合并(8px)
│   ├ 端口→节点: 300px半径 + 500px回退
│   ├ 端口→端口直连: 对齐200px / 普通100px
│   ├ 节点→节点: 对齐40px(500px距离) / 近邻80px
│   └ 防短路: 同元件不同端口不连同一节点
│
├─ Step 8: Manhattan布线
│   ├ L型: 先水平后垂直, 或先垂直后水平
│   ├ 避障: 6px扩展边距, Cohen-Sutherland裁剪
│   └ 失败跳过
│
├─ Step 9: 连接图构建
│   └ 并查集合并端口 → 连通组 + 端口标签
│
├─ Step 10: LLM评价
│   ├ 结构化提示词: 合规检查/DFMEA/原理评估
│   └ 被过滤时短提示词回退
│
├─ Step 11: 绘制标注图
│   ├ 元件: 色框 + 位号=值
│   ├ 端口: 红点 + 极性标签 (A/K/+/-/GND)
│   ├ 节点: 绿点 + J1/J2编号
│   ├ 连线: 蓝线 (Manhattan路由)
│   └ OCR: 橙色框 + 原文
│
└─ Step 12: 保存TXT + 图片
```

### 4.2 输出文件

| 文件 | 内容 |
|------|------|
| `xxx_wired.jpg` | 标注图 (色框+红点+绿点+蓝线+标签) |
| `xxx_result.txt` | 结构化报告 (元件清单+节点坐标+连接关系+LLM评价) |

### 4.3 关键规则

**端口位置** (归一化坐标, 0~1):
| 元件 | 默认端口 | 说明 |
|------|---------|------|
| R/C/L/D | (0,0.5),(1,0.5) | 左右 |
| V-DC/V-AC | (0.5,1.0),(0.5,0.0) | 下-/上+ |
| GND | (0.5,0.0) | 单端向上 |
| BJT | (0,0.5),(0.7,1.0),(0.7,0.0) | B/C/E |
| Op-Amp | (0,0.5),(0,0.3),(1,0.5) | -/+/OUT |
| IC | (0,0.3),(0,0.7),(1,0.5) | -/+/OUT |
| Voltage-Regulator | (0,0.5),(1,0.5),(0.5,0.0) | IN/OUT/GND |

**OCR改正记忆** (`ocr_corrections.json`):
```json
{
  "50H": "50kΩ",
  "400V": "400Ω"
}
```
手动添加常见CRNN错误 → 下次运行自动修正。

## 5. 文本评价流水线

### 5.1 方案对比

| 方案 | 入口 | 方法 | 特点 |
|------|------|------|------|
| Phase 1 (B) | `phase1_verify/evaluate.py` | 文字→LLM直接评价 | 最简, JSON输出 |
| Phase 2 (A) | `phase2_pipeline/report.py` | 文字→结构化→规则+LLM | 安全/正确性/质量三维 |
| Phase 3 | `phase3_spice/simulate.py` | 结构化→SPICE→仿真 | 需要Ngspice |
| Phase 4 (D) | `phase4_netlist/evaluate.py` | 文字→LLM→SPICE→解析 | 网表优先 |

### 5.2 Phase 2 评分公式

```
总分 = 100 - 致命错误×30 - 主要问题×15 - 次要问题×5 - 质量问题×3
     (下限0, 上限100)
```

**致命错误**: 电源短路、极性反接、回路断开  
**主要问题**: 缺失元件、拓扑错误、参数不合理  
**次要问题**: 多余元件  
**质量问题**: 参数优化、功耗、鲁棒性、成本

## 6. LLM集成

### 6.1 API配置

```
提供商: DeepSeek (默认) 或 Anthropic Claude
API端点: https://api.deepseek.com/v1 (OpenAI兼容)
模型: deepseek-v4-pro
Temperature: 0.0
Max Tokens: 2048
重试: 最多3次 (原提示词→短提示词回退)
```

### 6.2 环境变量

```bash
ANTHROPIC_API_KEY=sk-xxx    # Claude API Key
OPENAI_API_KEY=sk-xxx       # DeepSeek/OpenAI Key
ANTHROPIC_AUTH_TOKEN=sk-xxx # 备选Token名
ANTHROPIC_BASE_URL=https://api.deepseek.com/anthropic  # 自定义端点
```

### 6.3 LLM 评价结构

```
## 一、合规性/规则检查 (Rule-Based Audit)
- 电源与地: 短路/反接/GND悬空
- 元件参数: 数值范围合理性
- 极性元件: 方向正确性
- 回路完整性: 闭合/悬空节点

## 二、设计缺陷与风险分析 (DFMEA)
- 失效模式: 开路/短路/过流/过压/反压
- RPN = 严重度(S) × 发生度(O) × 探测度(D)
- TOP3风险项

## 三、经验性原理评估 (Heuristic Review)
- 电路类型推测
- 参数匹配性 (RC常数, 分压比, 限流等)
- 工程实践建议

## 总结
- 质量评分 (1-10)
- 改进建议
```

## 7. 运行方式

### 图像评价

```bash
# 单张图片
python -m src.vision.unified_pipeline circuit_5.jpg

# 多张图片
python -m src.vision.unified_pipeline text1.jpg text2.jpg text3.jpg

# 所有图片 (默认匹配 circuit_*.jpg)

```

### 文本评价

```bash
# Phase 1: 直接LLM评价
python -c "from src.phase1_verify.evaluate import evaluate_circuit; print(evaluate_circuit('R1=220Ω串联LED到5V'))"

# Phase 4: SPICE网表优先
python eval.py --sim data/samples/circuit_01_good.txt

python eval.py --all --fast
```


## 9. 常见问题

### OCR识别错误

**现象**: `50kΩ`被识别为`50H`, `2.2kΩ`被识别为`22Ω`

**原因**: CRNN手写体字符混淆 (k↔H, Ω↔V, 小数点丢失)

**解决**: 
1. 后处理variant系统自动修正大部分常见混淆
2. 手动添加到`ocr_corrections.json`
3. 使用机画OCR模型 (切换到`runs/ocr_crnn_machine/best.pt`)

### LLM评价被过滤

**现象**: 某些图片的评价为空或截断

**原因**: DeepSeek内容过滤器对含电路元件词的提示词触发不一致

**解决**: 系统自动回退到短提示词模式 (132字符), 返回精简评价

### 节点太少/连线不完整

**现象**: text1只有2个节点, 部分端口未连线

**原因**: CGHD模型对手绘/简单电路的节点检测率低

**解决**: 已启用端口直连回退 (对齐200px内), 但受限于物理距离

### IC型号读不出来

**现象**: `SGM2212-3.3` 被CRNN读成乱码

**原因**: CRNN字符集只有数字+单位, 不认识字母

**解决**: 系统自动用Tesseract读取IC附近文字, 优先选最长字母数字串

### GND符号有多个

**现象**: 连接关系中出现GND重复

**原因**: 电路图中有多个GND符号 (电气上连通), 每个都被单独检测

**解决**: 多GND符号是正常的, 连接图会自动合并到同一连通组
